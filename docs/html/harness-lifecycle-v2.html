Retention is not recovery. Recovery is not replay.

Split broker lifecycle into three policies with separate safety rules.

Adopt idle self-retirement and harness generation fencing now. Treat child restart as process recovery. Keep automatic user-turn retry disabled unless a driver can prove the previous attempt did not finish and did not mutate the outside world.

00 Policy placement is the contract risk

Correction

Policy placement is the main fix

HRC may own runtime lifecycle policy, but the compiled start request remains ASP-owned. The proposal moves lifecycle policy into a typed BrokerLifecyclePolicyOverlay on InvocationDispatchRequest, persisted and hashed separately from the compiled start request.

Why this matters

Lifecycle ambiguity currently leaks across planes

  • Broker-tmux runtimes can remain resident indefinitely.
  • Hung or dead harness children fall to coarse HRC stale/zombie recovery.

  • No protocol field currently identifies harness generation or turn attempt.

01 HRC owns hard control; broker owns soft lifecycle

The boundary is a safety contract. HRC owns durable runtime authority and public lifecycle semantics. The broker owns live harness-child mechanics below the broker boundary. ASP remains the compiler for execution mechanics.

ASP compiler plane

Compiles execution mechanics

  • Emits compiled plan and selected profile inputs.
  • Owns argv/env/cwd, driver spec, continuation encoding, and hashes.
  • Must recompile if lifecycle policy changes execution mechanics.
->
HRC control plane

Owns durable authority

  • Owns runtime container, broker process, tmux lease, durable state, public lifecycle policy, reconciliation, and hard reap.
  • Supplies an admitted terminal lease for tmux routes.
  • Persists policy hashes and terminal projections transactionally.
->
Broker execution plane

Owns live harness control

  • Owns native protocol handling, child supervision, recycle, input disposition, permission requests, and normalized events.
  • May use driver-private quit text or runner control channels when policy and capability allow.
  • Must not allocate or mutate tmux lifecycle objects.

What to notice: the arrows are ownership handoffs, not permission to rebuild lower-plane facts. ASP compiles mechanics, HRC admits and persists policy, and the broker emits live harness facts.

"HRC owns the process"

Precise split

HRC can stop, dispose, reconcile, and hard-reap. The broker supervises the harness child and emits the lifecycle facts HRC projects.

02 Keep HRC policy outside the compiled start request

The overlay preserves compiler closure. It is typed policy, not environment metadata. It is carried beside the compiled start request, validated against lifecycle capabilities, and audited with its own canonical hash.

Envelope location
InvocationDispatchRequest.lifecyclePolicy, outside startRequest.
Audit identity
policyId plus policyHash, persisted before broker invocation starts.
Hash rule
startRequestHash remains the ASP-compiled start request hash. Lifecycle policy changes do not mutate it.
Fallback rule
No silent downgrade from idle-ttl, recycle-child, or guarded retry unless HRC explicitly requested fallback behavior.

What to notice: the overlay has its own identity and audit path. If policy changes argv, env, cwd, driver config, or continuation encoding, it is no longer an overlay and must be recompiled by ASP.

export interface InvocationDispatchRequest {
  startRequest: InvocationStartRequest
  dispatchEnv?: Record<string, string>

}

export interface BrokerLifecyclePolicyOverlay {
  schemaVersion: 'harness-broker.lifecycle-policy/v1'
  policyId: string
  policyHash: string
  retention: RuntimeRetentionPolicy
  harnessRecovery: HarnessRecoveryPolicy
  turnRetry: TurnRetryPolicy
}

Contract shape: startRequest stays byte-stable while lifecycle policy travels as an adjacent typed control-plane field.

03 Separate residency, recovery, and replay

The three policies answer different questions. Keeping them separate prevents a healthy process recovery feature from becoming an unsafe user-input replay system.

R
RuntimeRetentionPolicy

Between-turn residency

Controls whether a healthy invocation stays alive or self-retires after it is idle.

  • Modes: keep-alive, idle-ttl, transitional unmanaged.
  • Idle clock starts only in ready, with no active turn, no pending permission, and an empty input queue.
  • Retirement emits terminal lifecycle events and no user-turn accounting.
G
HarnessRecoveryPolicy

Generation-aware child recovery

Controls observation and repair of the harness child process inside one broker invocation.

  • Modes: none, observe, recycle-child.
  • Uses harness generations to fence stale events, hooks, and permissions.
  • Stall detection must combine no progress with a health signal.
T
TurnRetryPolicy

Semantic replay, fail closed

Controls whether an accepted user input can be replayed after a failed attempt.

  • Default: mode:'none'.
  • safe-retry requires no tool call, no pending permission, no assistant final, no mutation evidence, known continuation, and proof the prior turn is incomplete.
  • Retry keeps the same inputId and logical turnId, then increments turnAttempt.

Invocation state remains container-level. Harness generation state is nested under it, so child recycle can repair a generation while the runtime lease and invocation identity remain stable.

Clean self-retirement is a broker terminal fact

Idle clock expires

Driver-private retire runs

The broker emits harness.exited and invocation.exited{reason:'idle-ttl'} after child or runner retirement completes.

Recycle flow

Process recovery does not replay input

Generation N fails

A child exits while ready, or a turn stalls with both no progress and a failed health signal.

Broker fences old generation

Old-generation events become diagnostics only; pending permissions are cancelled with generation and attempt identity.

Generation N+1 starts

The runtime lease and invocation identity remain unchanged. The broker emits harness.started for the new generation and returns to ready if recovery succeeds.

05 Generation-aware events make recovery auditable

Lifecycle must be observable as first-class broker facts. Existing events get narrow extensions; lifecycle policy and generation data do not belong inside invocation.started.

lifecycle.policy.accepted

Records accepted policy hash and modes after broker validation.

lifecycle.escalation

Requests HRC hard reap or operator attention after broker-side recovery cannot proceed.

harness.started

Starts generation 1 or a recycled generation with mechanism and optional pid/control socket evidence.

harness.exited

Terminates a child generation without automatically terminating the invocation.

harness.recovery.started

Captures child-exit, stall, or healthcheck-failed recovery reasons and active turn disposition.

harness.recovery.completed

Advances the active harness generation when recycle succeeds.

turn.stalled

Reports threshold, no-progress window, health probe, generation, and attempt.

turn.retry

Audits a safe retry under the same logical input and turn, with incremented attempt.

permission.cancelled

Cancels old-generation permission requests without overloading allow | deny.

Scan path: policy events explain what was accepted, harness events explain which child is current, turn events explain attempt semantics, and permission cancellation prevents stale approvals from crossing generations.

Avoid

Adding cancelled to permission decisions

The current permission decision contract is only allow | deny. Overloading it would make stale generation handling ambiguous.

use
Required

permission.cancelled plus composite identity

Persist uniqueness by globally unique permission IDs or by (invocation_id, harness_generation, turn_attempt, permission_request_id).

06 Projection must fence stale facts

HRC's broker event mapper is the only place broker events become runtime/run/message state. Projection must enforce sequence idempotence, generation equality, and clean terminal ordering.

lifecycle.policy.accepted

Store accepted policy hash and modes on broker invocation and runtime state.

harness.started

Set current harness generation and record the generation transition while runtime remains active.

turn.retry

Add an attempt record under the same logical input and turn. Do not create a second accepted input.

invocation.exited{reason:'idle-ttl'}

Project clean runtime termination before liveness reconcile can classify the pane as stale.

permission.cancelled

Mark pending permission stale/cancelled and block late decisions from reaching the current generation.

lifecycle.escalation

Record broker-requested escalation, then let HRC hard-reap as a control-plane action.

Projection rule: event idempotence by (invocationId, seq) is necessary but not sufficient; state-affecting projection also needs generation and attempt checks.

08 Truthful failure beats hidden replay

Recovery should preserve truthful state. It should not hide uncertainty by converting a stalled or crashed turn into an unproven successful replay.

Clean

Idle retire succeeds

Broker emits invocation.exited{reason:'idle-ttl'}. HRC treats pane disappearance as cleanup confirmation, not stale evidence.

Recoverable

Child crashes while ready

Broker recycles if policy allows, otherwise fails or escalates. Runtime remains active only if recovery completes.

No replay

Turn stalls

Broker emits turn.stalled, then fails active turn or escalates. Replay requires proven safe-retry.

Escalate

Runner unresponsive

Broker emits lifecycle.escalation{requestedAction:'hard-reap'}. HRC records the request and hard-reaps.

Conservative

Broker process dies

No broker event exists. HRC uses stale/orphan reconciliation, not semantic retry.

Fence

Late old-generation hook

Store as diagnostics only. Do not project into active runtime, turn, permission, assistant, or tool state.

09 Tests decide when capability can be advertised

The contract is complete only when these checks exist. They are framed as visible gates rather than prose preferences.

Compiler closure and policy transport

Prove lifecycle overlay does not mutate startRequest bytes/hash; reject attempts to patch lifecycle into HarnessInvocationSpec; persist policy hash separately.

Idle TTL

Verify ready/empty queue/no permissions starts the timer; active turn, queued input, or pending permission prevents it; clean retire emits no input.accepted.

Harness generations

Assert initial generation 1, recycle generation 2, stable runtime lease, exactly-once generation advancement, and old-generation event rejection.

Process-tree correctness

Kill/recycle children with descendants, escalate stubborn children, handle runner death during recycle, restore terminal foreground, and prove no forbidden tmux lifecycle commands.

Turn retry

Default retry disabled; retry emits no second input.accepted; unsafe evidence suppresses retry; exhausted retry fails the turn terminally.

Permission fencing

Requests include generation and attempt; recycle emits permission.cancelled; late old-generation decisions are rejected or audited.

10 Ship cleanup before recycle, retry last

The migration sequence protects semantics while still delivering value early. It keeps high-value, low-risk lifecycle cleanup ahead of process recycle, and keeps semantic retry last.

0

Spec and data model only

Add lifecycle overlay, capability fields, harness generation, turn attempt, lifecycle events, HRC persistence, and normalizer preservation. No driver advertises capability yet.

1

Idle TTL only

Implement real driver-private retire, broker idle timer, clean terminal events, HRC clean projection, and reconcile race fencing. This is the first production value slice.

2

Harness generations and recovery without retry

Add runner control channel, process-tree recycle, hook socket rotation or equivalent generation fence, turn stall reporting, fail-before-recycle, and return to ready.

3

Guarded retry for proven-safe subset

Implement retry predicates, fail-closed behavior, attempt projection, and enable only for drivers that can prove no prior completion and no side effects.

4

Long-term route cleanup

Prefer real non-interactive or SDK-backed drivers for Discord/ariadne-style workloads. Keep tmux where operator-visible interactive sessions are the right substrate.

Contract verdict

Adopt the proposal with the corrections below. Without them, the contract blurs compiler closure, overstates retry safety, under-models generation identity, and risks converting clean lifecycle events into stale runtime failures.

  • Use three policies: retention, harness recovery, and turn retry.
  • Carry HRC-authored policy as a separately hashed dispatch overlay, or require ASP to compile it.
  • Keep child recycle as process recovery, not automatic user-input replay.
  • Add harness generations, turn attempts, permission cancellation, and transactionally clean idle projection.
