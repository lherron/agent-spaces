ASP compiler plane -> HRC control plane -> Harness Broker execution plane

Close the runtime plan before HRC touches it.

The rewrite turns Agent Spaces into a real compiler for runtime execution. It emits immutable, hashable runtime plans; HRC admits and controls a selected profile; the broker executes the ASP-emitted start request unchanged.

Compile
ASP resolves runtime construction
Placement, bundle, harness runtime, model, prompt materialization, env, process argv, driver spec, continuation, hashes.
Select
HRC routes over profiles
Route decisions select one profile by profileId and profileHash; they do not derive hidden execution profiles.
Execute
Broker owns harness protocols
The broker consumes InvocationStartRequest, drives native harnesses, and emits normalized ordered events.
Load-bearing invariant
After ASP emits a CompiledRuntimePlan, HRC must not reconstruct, mutate, patch, infer, or synthesize broker execution mechanics.

01 Semantic Shifts

The meaningful change is not only new names. It moves responsibility for execution facts out of HRC and makes runtime selection a typed, audited decision over compiler products.

ASP
From helper API to compiler plane

ASP owns reproducible runtime construction and emits the single immutable product: CompiledRuntimePlan.

HRC
From branch mesh to profile selection

HRC chooses a profile, validates policy and capabilities, persists evidence, then delegates mechanics to a controller.

Broker
From wrapper process to execution plane

Harness Broker owns native driver protocols, input disposition, permission requests, and normalized event streams.

Legacy
exec.ts stops being a boundary

Legacy launch wrappers can exist only as feature-gated migration machinery, not as a place for new harness semantics.

As-is

Execution facts are assembled late

HRC launch artifacts, exec.ts, Codex launch detection, spool/callback paths, and native event parsing are entangled in the control plane.

replace with
To-be

Execution facts are compiled once

ASP emits complete execution profiles. HRC persists redacted artifacts and sends the selected broker start request unchanged.

02 Contract Planes

The architecture reads as a triangle of ownership, not a wrapper stack. Each plane has an explicit product and forbidden knowledge.

ASP compiler plane

Owns construction

  • Placement, bundle resolution, target materialization
  • Harness family/runtime and model resolution
  • Process command, args, cwd, env
  • Driver spec, transport descriptor, continuation encoding
  • Redacted artifacts and semantic hashes
->
HRC control plane

Owns admission and lifecycle

  • Identity allocation and operation records
  • Route admission and selected profile
  • Reuse, policy, persistence, public API views
  • Runtime state, reconciliation, stop, dispose
  • Permission mediation and event projection
->
Broker execution plane

Owns harness execution

  • Broker process and driver registry
  • Native harness protocol parsing
  • Invocation start, input, interrupt, stop, status
  • Permission requests and normalized event stream
  • Driver continuation reports and redaction

03 Control Flow

The first concrete target is Codex headless through the broker. This flow is the executable reading of the compiler closure invariant.

HRC allocates identities before compilation

Request, operation, runtime, invocation, input, run, and trace IDs enter the compile request so hashes and diagnostics can be correlated end to end.

HRC calls compileRuntimePlan(req)

The compile input is the only place HRC can influence broker execution mechanics. Later policy changes require rejection or recompilation.

ASP emits a closed plan

CompiledRuntimePlan contains one or more complete RuntimeExecutionProfile values, including a broker profile for admissible Codex headless routes.

HRC selects and validates one profile

RuntimeRouteDecision references compileId, planHash, selectedProfileId, and selectedProfileHash.

Broker controller passes start request unchanged

HarnessBrokerController starts and supervises the broker process, validates hello/status, then sends ASP's InvocationStartRequest.

Events become HRC state through one mapper

Broker events are persisted by (invocationId, seq) and projected idempotently into runtime, run, message, tool, usage, continuation, and permission records.

04 Datatype Surface

The supporting datatype rewrite creates a shared DTO package and names the contract records that carry compiler closure across package boundaries.

Package

packages/spaces-runtime-contracts

Shared IDs, compiler plan, execution profiles, capabilities, route decisions, runtime state, operations, policy, continuation, redaction, hashes, public views, and persistence DTOs.

It must not import HRC server code, broker driver code, or concrete harness implementation packages.

API

compileRuntimePlan(req)

Agent Spaces owns the compiler API and returns either a complete plan or structured redacted diagnostics. HRC broker-capable routes call this API instead of building broker invocations directly.

RuntimeCompileRequest
HRC -> ASP

Identity allocation, placement, requested model/harness/runtime, materialization inputs, HRC policy overlays, continuation, and correlation.

CompiledRuntimePlan
ASP product

Compiler metadata, plan hashes, identity, placement, resolved bundle, harness/model resolution, profiles, artifacts, secrets, diagnostics.

RuntimeExecutionProfile
Closed strategy

Union of terminal, embedded SDK, harness broker, command process, and legacy execution profiles. Each profile is complete or invalid.

BrokerExecutionProfile
Codex headless target

Broker protocol, driver, ownership, invocation start request, spec/start hashes, redacted artifacts, policy, continuation, and observability.

RuntimeRouteDecision
HRC route product

Selects one compiled profile by ID/hash, names controller, records admission, reuse, product policy, capability resolution, and legacy alias.

RuntimeState
Controller-discriminated

Typed runtime metadata replaces mismatched optional blobs. Broker state carries compile, profile, spec, invocation, capability, continuation, permission, and input state.

RuntimeCompileRequest

type RuntimeCompileRequest = {
  schemaVersion: 'agent-runtime-compile-request/v1' // Selects the compile request wire contract.

  identity: { // HRC-allocated IDs that make the compiled plan auditable and replayable.
    requestId: RequestId // Correlates this compiler call with the original HRC request.
    operationId: RuntimeOperationId // Names the HRC control-plane operation being compiled.
    hostSessionId: HostSessionId // Identifies the user-visible session or conversation continuity.
    generation: number // Guards against stale runtime/session callbacks after rotation.
    runtimeId: RuntimeId // Names the reusable runtime HRC will control if admitted.
    invocationId?: InvocationId // Preallocates broker invocation identity for broker profiles.
    initialInputId?: InputId // Preallocates the first input ID when an initial prompt exists.
    runId?: RunId // Preallocates the user-visible run or turn identity.
    traceId?: TraceId // Carries distributed trace correlation across planes.
    idempotencyKey?: string // Lets HRC retry compilation without creating a different semantic request.
  }

  placement: { // Requested workspace/target placement; ASP resolves the bundle from here.
    kind?: string // Placement strategy or shape identifier from the ASP request/registry.
    root?: string // Filesystem or workspace root used to resolve the runtime target.
    targetName?: string // Logical target name when placement points at an ASP target.
    targetDir?: string // Concrete target directory when already resolved or explicitly supplied.
    [key: string]: unknown // Extension fields preserved for placement implementations outside this DTO.
  }

  requested: { // User or route preference before ASP resolves concrete runtime facts.
    modelProvider?: ProviderDomain // Preferred model provider family, such as anthropic or openai.
    model?: string // Requested model alias or provider model ID before compiler normalization.
    reasoningEffort?: 'low' | 'medium' | 'high' | 'xhigh' // Requested reasoning tier when supported.
    harnessFamily?: HarnessFamily // Requested harness family, such as claude-code, codex, or pi.
    preferredHarnessRuntime?: HarnessRuntime // Preferred materialized runtime, such as codex-cli or pi-sdk.
    interactionMode?: InteractionMode // Requested interaction shape: interactive, headless, or nonInteractive.
  }

  materialization: { // Inputs ASP may turn into files, env, prompt material, or broker input.
    initialPrompt?: string // Initial user prompt or launch input to compile into the selected profile.
    attachments?: AttachmentRef[] // Local files/images/opaque refs ASP validates and materializes.
    taskContext?: HrcTaskContext // Task metadata, role, evidence requirements, and hints from HRC.
    resolvedBundleHint?: ResolvedRuntimeBundle // Optional prior bundle resolution to reuse or validate.
  }

  hrcPolicy: { // HRC product/security overlays that ASP compiles into explicit profile policy.
    permissionPolicy?: BrokerPermissionPolicy // Permission posture: deny, allow with provenance, or ask-client.
    inputPolicy?: BrokerInputPolicy // Runtime input behavior such as ready input and busy handling.
    exposurePolicy?: AgentchatExposurePolicy // Whether the runtime is exposed to Agentchat targets.
    resourceLimits?: RuntimeResourceLimits // Startup, turn, stop, queue, event, and age limits.
    observability?: RuntimeObservabilityInput // Trace and telemetry inputs that may affect env/driver config.
    capabilityPolicy?: HrcCapabilityPolicy // Whether degradation is allowed and which capabilities are required.
  }

  continuation?: { // Existing HRC/broker continuation key to encode into the new profile.
    schemaVersion: 'runtime-continuation/v1' // Selects the continuation reference contract.
    hrc: { // HRC-owned continuation identity used for product/API compatibility.
      provider: ProviderDomain // Provider that owns the continuation key, such as anthropic or openai.
      keyHash: string // Stable redacted hash for persistence, matching, and diagnostics.
      key?: string // Raw key only while needed in memory; persisted artifacts use hashes/digests.
    }
    broker?: { // Broker/native continuation identity when the execution plane has its own key.
      provider: string // Broker or driver provider name that produced the continuation.
      kind?: 'thread' | 'session' | 'conversation' | string // Native continuation category.
      keyHash: string // Stable redacted broker continuation hash.
      key?: string // Raw broker key only while needed to resume or dispatch.
    }
    source: 'embedded-sdk' | 'harness-broker' | 'legacy-exec' | 'terminal-hook' // Where the continuation was observed.
    sourceEvent?: { // Event evidence tying the continuation back to a broker/controller fact.
      invocationId?: InvocationId // Broker invocation that reported the continuation.
      eventSeq?: number // Ordered broker event sequence where the continuation was seen.
      eventType?: string // Event type that carried the continuation payload.
    }
    observedAt: IsoTimestamp // When HRC observed or persisted this continuation reference.
  }

  correlation: { // Non-authoritative correlation copied into diagnostics, artifacts, and broker observability.
    requestId: RequestId // Repeats request correlation for consumers that do not inspect identity.
    operationId?: RuntimeOperationId // Optional operation correlation for follow-up inputs or inspections.
    hostSessionId: HostSessionId // Session continuity for logs, public views, and artifacts.
    generation: number // Generation correlation used during stale callback/reconcile checks.
    runtimeId?: RuntimeId // Runtime correlation when the target runtime already exists.
    runId?: RunId // Run correlation when this compile belongs to a user-visible turn.
    invocationId?: InvocationId // Broker invocation correlation when one is already allocated.
    traceId?: TraceId // Distributed trace ID for observability backends.
    appId?: string // Optional application/client identifier for public API consumers.
    appSessionKey?: string // Optional app-specific session key for client-side grouping.
    scopeRef?: string // Optional agent scope reference, such as cody@agent-spaces.
    laneRef?: string // Optional parallel lane identifier within the scope.
  }
}

RuntimeCompileResponse

The immutable compiler product HRC may select, persist, reject, or send unchanged to the broker.

type RuntimeCompileResponse =
  | {
      schemaVersion: 'agent-runtime-compile-response/v1' // Selects the compile response wire contract.
      ok: true // Means ASP produced a complete plan HRC may route over.
      plan: CompiledRuntimePlan // The closed compiler product; HRC must not fill in missing mechanics.
      diagnostics: CompileDiagnostic[] // Redacted warnings/info that remain useful even when compilation succeeds.
    }
  | {
      schemaVersion: 'agent-runtime-compile-response/v1' // Same response schema for failed compilation.
      ok: false // Means HRC must reject or recompile; it cannot patch missing mechanics.
      diagnostics: CompileDiagnostic[] // Redacted errors/warnings explaining why no usable plan was emitted.
    }

type CompiledRuntimePlan = {
  schemaVersion: 'agent-runtime-plan/v1' // Selects the immutable runtime plan schema.
  compiler: { // Identifies the compiler implementation that produced the plan.
    name: 'agent-spaces' // Compiler owner; keeps the plan attributable to ASP.
    version: string // Compiler version used for diagnostics, hashing, and migration checks.
  }
  compileId: CompileId // Unique compiler attempt ID for artifacts, route decisions, and operations.
  planHash: PlanHash // Semantic hash of the complete plan with raw secrets removed or digested.
  redactedPlanHash: RedactedPlanHash // Hash of the exact redacted artifact HRC may persist.
  createdAt: IsoTimestamp // Compile timestamp; audit evidence excluded from semantic hashes.

  identity: RuntimeIdentityAllocation // Echoes HRC-allocated IDs embedded into profiles and start requests.
  placement: { // Placement request ASP compiled from.
    kind?: string // Placement strategy or shape identifier from the ASP request/registry.
    root?: string // Filesystem or workspace root used to resolve the runtime target.
    targetName?: string // Logical target name when placement points at an ASP target.
    targetDir?: string // Concrete target directory when already resolved or explicitly supplied.
    [key: string]: unknown // Extension fields preserved for placement implementations outside this DTO.
  }
  resolvedBundle: ResolvedRuntimeBundle // Concrete bundle/materialization result ASP selected.

  harness: { // Resolved harness identity after ASP catalog and placement decisions.
    family: HarnessFamily // Harness family, such as codex, pi, or claude-code.
    runtime: HarnessRuntime // Concrete runtime implementation, such as codex-cli or claude-agent-sdk.
    provider: ProviderDomain // Model provider domain for credentials and continuation compatibility.
  }

  model: { // Resolved model selection that the selected runtime will use.
    provider: ProviderDomain // Provider domain after normalization.
    modelId: string // Authoritative provider model ID emitted by ASP.
    requestedModel?: string // Original user/API model value when it differs from modelId.
    reasoningEffort?: 'low' | 'medium' | 'high' | 'xhigh' // Resolved reasoning tier when present.
  }

  executionProfiles: RuntimeExecutionProfile[] // Complete strategies HRC can select; HRC may not complete them.

  artifacts: { // File/bundle artifacts that let HRC persist evidence without owning construction.
    materializedBundleRoot?: string // Filesystem root of the resolved runtime bundle.
    systemPromptFile?: string // Materialized system prompt path when ASP writes one.
    userPromptFile?: string // Materialized user prompt path when ASP writes one.
    lockHash?: string // Bundle lock hash used to reason about reproducibility.
    bundleIdentity: string // Stable identity of the resolved bundle.
  }

  secrets: { // Secret classification evidence without exposing raw secret values.
    envKeys: string[] // Environment keys present in the executable request.
    secretEnvKeys: string[] // Environment keys classified as secrets and excluded from persisted artifacts.
    secretDigests?: Record<string, SecretDigest> // Stable scoped digests for secret-aware hashes.
  }

  diagnostics: CompileDiagnostic[] // Plan-scoped compiler facts, warnings, or errors.
}

type CompileDiagnostic = {
  level: 'info' | 'warning' | 'error' // Severity HRC uses for admission and public warnings.
  code: string // Stable machine-readable diagnostic code.
  message: string // Human-readable redacted explanation for logs and review.
  plane: 'asp-compiler' // Names the source plane; compiler diagnostics are not runtime events.
  profileId?: ProfileId // Optional profile-specific target for the diagnostic.
  redactedDetails?: unknown // Structured details with secrets removed or replaced by digests.
}

06 Rules That Matter

These are the review rules that should be visible in any implementation plan. They are where the compiler rewrite prevents the old architecture from reappearing under new names.

HRC may select

HRC can select one profile already present in the compiled plan, reject it, persist redacted artifacts, validate capabilities, and ask ASP to recompile.

HRC must not complete

HRC cannot create or mutate HarnessInvocationSpec, InvocationStartRequest, driver config, process argv/env/cwd, or prompt materialization.

Policy changes recompile

If placement, model, harness, continuation, permissions, prompt materialization, environment, or driver requirements change, the answer is recompile, not patch.

Forbidden for broker routes

HRC invoking launch/exec.ts

No broker-capable path may shell through a launch artifact, rely on callback/spool delivery as the event bus, or parse native harness output.

enforce via
Boundary checks

Static checks plus route tests

CI should reject concrete harness driver imports, broker spec construction/mutation in HRC paths, and any Codex broker route that spawns legacy execution.

07 Review Gates

This first pass treats these as acceptance gates for the compiler rewrite slice, not for the entire runtime-control refactor.

Plan completeness ASP emits versioned, hashable, redacted CompiledRuntimePlan artifacts for supported runtime shapes.
Broker profile completeness Codex headless emits a complete BrokerExecutionProfile with start request, hashes, policy, capabilities, continuation, and observability.
Compiler determinism Plan, profile, compatibility, spec, and start-request hashes are stable and change when process or driver mechanics change.
Secret discipline Redacted plans/specs/start requests contain no raw secrets; persisted secret evidence uses scoped digests or secret refs.
Route selection HRC route decisions select execution profiles from compiled plans and derive legacy transport aliases only for public compatibility.
Boundary proof HRC broker paths do not import driver internals, parse native harness events, spawn exec.ts, or assign broker spec/process fields.

08 Review Pressure Points

These are the architectural questions that stand out from the first pass and should be resolved before the UI/UX or remaining refactor pages are pulled in.

Does HRC need any non-mechanical overlay after compilation?

The contracts allow explicit HRC overlays only when they do not mutate process or driver mechanics. That boundary should be concrete in code and tests.

Is RuntimeExecutionProfileKind = RuntimeControllerKind intentional long-term?

The final datatypes make profile kind and controller kind share values such as terminal and harness-broker. That is tidy, but it may constrain future cases where one controller can execute multiple profile families.

Where does the final permission event vocabulary freeze?

The contract notes that invocation.permission.request should be normalized or explicitly versioned as a broker-to-client request method. This should settle before protocol freeze.

How hard is the legacy deletion line?

The deletion criterion is strong: no broker-capable harness execution path in HRC may depend on exec.ts, launch artifacts, callback/spool delivery, native protocol parsing, or HRC reconstruction.
