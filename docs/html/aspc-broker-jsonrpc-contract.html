ASP compiler plane -> Harness Broker execution plane

JSON-RPC contract for compile and runtime control.

ASPC exposes compiler products over aspc/0.1. Harness Broker exposes runtime control over harness-broker/0.1 and harness-broker/0.2. The boundary rule stays the same: ASP compiles, HRC selects and dispatches, broker executes.

Compiler facade
aspc.*
Compile runtime plans, select broker profiles, optionally compile and start through a co-hosted broker facade.
Broker runtime
broker.* and invocation.*
Negotiate capabilities, start compiled invocations, route input/control, attach after reconnect, replay events, and resolve permissions.
Shared object handoff
InvocationDispatchRequest
The selected compiled InvocationStartRequest travels unchanged, with dispatch/runtime/lifecycle overlays outside hash material.

01 JSON-RPC Envelope

Both protocol namespaces use JSON-RPC 2.0 request/response messages. Broker runtime also uses JSON-RPC notifications for event delivery.

Copied from the current broker JSON-RPC transport shape, which the ASPC facade imports for request typing.

export type JsonRpcId = string | number | null // Request/response correlation token; null is allowed by JSON-RPC 2.0.

export interface JsonRpcRequest<TMethod extends string = string, TParams = unknown> { // Client-to-server command envelope.
  jsonrpc: '2.0' // Fixed JSON-RPC protocol version marker.
  id: JsonRpcId // Correlates the response with this request.
  method: TMethod // Fully-qualified method name, such as aspc.hello or invocation.start.
  params?: TParams | undefined // Method-specific request payload; omitted only when the method permits no params.
}

export interface JsonRpcNotification<TMethod extends string = string, TParams = unknown> { // One-way JSON-RPC message with no response.
  jsonrpc: '2.0' // Fixed JSON-RPC protocol version marker.
  method: TMethod // Notification method name.
  params?: TParams | undefined // Notification payload; broker events use this path.
}

export interface JsonRpcError { // Standard JSON-RPC error object.
  code: number // Numeric JSON-RPC error code.
  message: string // Human-readable error summary.
  data?: unknown // Optional structured diagnostic payload.
}

export interface JsonRpcResultResponse<TResult = unknown> { // Successful response envelope.
  jsonrpc: '2.0' // Fixed JSON-RPC protocol version marker.
  id: JsonRpcId // Echoes the request id.
  result: TResult // Method-specific success payload.
}

export interface JsonRpcErrorResponse { // Failed response envelope.
  jsonrpc: '2.0' // Fixed JSON-RPC protocol version marker.
  id: JsonRpcId // Echoes the request id when known.
  error: JsonRpcError // Structured failure payload.
}

export type JsonRpcResponse<TResult = unknown> = // Complete response union for request/response methods.
  | JsonRpcResultResponse<TResult> // Success branch.
  | JsonRpcErrorResponse // Error branch.

02 ASPC Methods

ASPC is additive compiler access. It returns the same compiler products as the SDK path and keeps broker runtime methods in the broker namespace.

Compiler facade

ASPC method surface

The compiler RPC surface is small on purpose: negotiate, compile a plan, select a broker-ready start request, or use the optional co-hosted convenience path.

aspc/0.1 compile start facade
Method Contract Purpose
aspc.hello paramsAspcHelloRequest resultAspcHelloResponse Negotiate aspc/0.1 and expose facade capabilities.
aspc.compileRuntimePlan paramsAspcCompileRuntimePlanRequest resultRuntimeCompileResponse Compile a closed CompiledRuntimePlan without selecting or starting a broker invocation.
aspc.compileHarnessInvocation paramsAspcCompileHarnessInvocationRequest resultAspcCompileHarnessInvocationResponse Compile, select an existing broker profile, and return its exact InvocationStartRequest plus dispatch wrapper.
aspc.compileAndStart paramsAspcCompileAndStartRequest resultAspcCompileAndStartResponse Convenience facade over compile/select followed by broker invocation.start when a broker is co-hosted.

03 Broker Methods

Broker methods control an already-compiled invocation. They do not compile, materialize, resolve placement, choose a harness, or prepare the compiled process shape.

Broker v1

V1 runtime surface

Baseline runtime control once HRC already has a compiled broker invocation request.

start input status
MethodPurpose
broker.helloProtocol/version/capability negotiation.
broker.healthHealth and optional driver probing.
invocation.startStart a compiled invocation from an InvocationDispatchRequest.
invocation.inputSubmit user input, context append, or steering correction.
invocation.interruptInterrupt a turn or invocation if supported by the driver.
invocation.stopRequest broker-controlled stop.
invocation.statusRead current invocation state.
invocation.disposeRelease broker state for an invocation.
Broker v2

V2 attach and replay additions

Reconnect-safe control for durable broker sessions, replayed events, and pending permission settlement.

attach replay permissions
MethodPurpose
broker.attachAttach a fresh controller to a broker invocation after reconnect/restart.
invocation.eventsSinceReplay ordered broker events after a sequence number, optionally live.
invocation.ackEventsAcknowledge durable event projection through a sequence number.
invocation.snapshotRead broker-owned invocation snapshot, including pending permissions.
invocation.permission.respondResolve a broker-to-client permission request.
Separated protocol namespaces

aspc.* methods are never part of harness-broker/0.1. A co-hosted ASPC+broker facade may multiplex both protocols on one transport, but method namespaces, validators, errors, and failure domains remain separate.
