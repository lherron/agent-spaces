Risks, Open Questions, and Implementation Plan

Risk register, open questions, phased implementation tasks, and definition of done.

Risks and mitigations

RiskImpactMitigation
Process startup overheadOne-shot broker adds a wrapper process and app-server/SDK startup remains expensive.Start with one-shot for reliability; add resident per-runtime mode for SDK/app-server where reuse matters. Avoid rematerialization in broker.
Event ordering and replayCallbacks/spool can duplicate or reorder events around process exit.Broker emits eventId and monotonic brokerSeq. HRC persists idempotently and applies generation fences.
Callback/spool reliabilityBroker may exit before events are delivered.Use file spool for detached/tmux one-shot; HRC owns replay. For direct headless, prefer stdio sink and synchronous HRC persistence.
Security/env leakageInherited HRC/Agent/Codex env can leak credentials or stale control paths.Default scrub inherited env; allow only explicit invocation env + explicit HRC childEnv. Redact logs. Validate cwd and executable.
PTY vs pipesSemantic events from interactive PTY are lossy.Declare event capabilities honestly. Use hooks/OTEL/app-server/SDK for structured events; do not pretend PTY transcript is a normalized event stream.
Provider continuation stateBad continuation after failure can poison later runs.Broker reports observed continuation. HRC persists only under success/accepted policy; retain current rule that failed SDK sessions do not propagate continuation.
In-flight input supportOnly some SDK paths support it today.Make in-flight input capability-driven. Use resident broker for Agent SDK. Reject unsupported semantics explicitly for Pi/CLI until proven.
Codex app-server special casesResume fallback, token usage, item mapping, final output synthesis are easy to duplicate incorrectly.Move existing runCodexAppServerOneShot integration into a single broker driver and test against golden event traces.
Claude/Pi parityCodex has richer headless/app-server paths; Claude/Pi may lag.Expose precise capabilities per driver. Do not leak provider-specific fallback logic into HRC.
Package boundary driftBroker may start importing compiler or HRC internals.Import-boundary tests and package-level dependency constraints.

Open questions

  1. Should HRC own tmux PTY forever? Recommendation: yes for initial broker; add broker-owned PTY only for non-tmux consumers.
  2. Where should Codex OTEL config injection live? Recommendation: compile/materializer or HRC prelaunch hook, not broker core. Broker can accept explicit driver env/config but should not know HRC auth.
  3. Should event sink be callback HTTP or stdout? Recommendation: support both. Use stdout for direct supervised headless, callback/spool for tmux/detached one-shot.
  4. Can Codex app-server support true multi-turn resident mode? Needs capability verification; protocol should support it without assuming it.
  5. How should HRC compatibility migration handle old event names? Add an HRC adapter translating broker events to current eventKind until consumers switch.
  6. Should HarnessInvocationSpec include top-level argv? Recommendation: no in final shape. Use driver.argv. Accept top-level argv only in compatibility parser.

Definition of done

  • hrc-server does not import Agent Spaces runtime APIs or harness driver internals.
  • All HRC launches use HrcBrokerLaunchEnvelope wrapping HarnessInvocationSpec.
  • Broker package imports no HRC packages.
  • Codex, Claude, Pi, Agent SDK, and Pi SDK each have declared capability fixtures.
  • Event replay is idempotent by eventId/brokerSeq.
  • HRC remains authoritative for run/runtime/session state, generation fences, persistence, and zombie handling.
