What the source currently does

Summary: the current implementation already has all three layers, but they are interleaved. Agent Spaces compiles and runs SDK sessions; HRC compiles, launches, persists, and contains harness routing; exec.ts is effectively a prototype one-shot broker with HRC-specific callback wiring.

Agent Spaces public surface mixes compile and runtime

agent-spaces/packages/agent-spaces/src/types.ts defines ProcessInvocationSpec with argv/env/cwd/io mode plus Codex app-server fields, and also exposes runtime execution and steering requests: RunTurnNonInteractiveRequest, QueueInFlightInputRequest, and InterruptInFlightTurnRequest. This confirms the public SDK currently spans both compile-time and invocation-time behavior.

  • types.ts:48-66 — current process invocation shape.
  • types.ts:88-153 — SDK turn and in-flight request shapes.
  • types.ts:158-186 — compile-time invocation request/response.
  • types.ts:320-326 — one client interface includes runtime and compile methods.

Compile path inside Agent Spaces

buildPlacementInvocationSpec resolves placement, plans runtime, detects the adapter, materializes bundles, materializes system prompts, prepares Codex homes, builds argv/env, prepares brain/tool env, and returns ProcessInvocationSpec. This is compiler/materializer work and should stay outside the broker.

  • client.ts:1005-1210 — placement compile/materialize path.
  • client.ts:1119-1130 — Codex runtime home preparation.
  • client.ts:1132-1157 — adapter argv/env construction and ASP env.
  • client.ts:1159-1179 — brain/tool runtime env preparation.
  • client.ts:1190-1204 — returns ProcessInvocationSpec and optional Codex app-server descriptor.

Runtime path inside Agent Spaces

runPlacementTurnNonInteractive does placement resolution and materialization again, then creates Agent SDK or Pi SDK sessions, tracks in-flight state, maps events, handles continuation, and returns a final result. These are broker/runtime-driver concerns, not compiler concerns.

  • client.ts:1217-1505 — placement-aware SDK runtime execution.
  • session-events.ts:26-67 — event emitter adds HRC-like hostSessionId/runId correlation.
  • session-events.ts:88-182 — maps unified SDK session events to Agent Spaces/HRC-ish event names.
  • run-tracker.ts:19-42 — in-flight run context stores session, continuation, outstanding turns, accepted input IDs.
  • run-tracker.ts:130-150 — queued in-flight prompts serialize over a session send chain.

Harness catalog and adapters are already in Agent Spaces

spaces-config defines the canonical harness catalog: Claude CLI, Claude Agent SDK, Pi CLI, Pi SDK, and Codex CLI. It also defines the current HarnessAdapter interface for detection, validation, materialization, composition, argv, env, and target bundle loading.

  • packages/config/src/core/types/harness.ts:61-97 — canonical harness catalog.
  • packages/config/src/core/types/harness.ts:468-540 — adapter contract.
  • packages/harness-codex/src/adapters/codex-adapter.ts:91-126 — Codex app-server launch descriptor builder.

HRC imports Agent Spaces runtime APIs and duplicates harness logic

hrc-server depends directly on agent-spaces, spaces-config, spaces-execution, spaces-harness-codex, and spaces-runtime. Its adapter layer repeats provider/frontend selection, env overlays, invocation mode decisions, brain/tool env prep, event translation, SDK turn routing, and in-flight delivery.

  • hrc-server/package.json — direct dependency on Agent Spaces runtime and harness packages.
  • agent-spaces-adapter/cli-adapter.ts:41-52 — provider/harness to frontend mapping duplicated in HRC.
  • cli-adapter.ts:173-227 — HRC-specific correlation env creation.
  • cli-adapter.ts:244-264 — HRC chooses headless vs interactive modes.
  • sdk-adapter.ts:114-165 — HRC calls queueInFlightInput in Agent Spaces.
  • sdk-adapter.ts:224-252 — HRC duplicates brain/tool env prep.
  • sdk-adapter.ts:307-377 — HRC runs SDK turns through Agent Spaces and maps events into HRC envelopes.

Current HRC launch artifact is overloaded

HrcLaunchArtifact combines durable HRC identity and callback configuration with the harness process invocation. That is exactly the split the new design should introduce.

  • hrc-core/src/contracts.ts:183-232 — launch artifact contains HRC IDs, callback/spool fields, argv/env/cwd, mode, prompts, hook bridge, Codex app-server, OTEL.
  • launch/launch-artifact.ts:6-20 — validator requires both HRC metadata and harness invocation fields together.

exec.ts is a prototype broker

hrc-server/src/launch/exec.ts already reads a launch JSON, validates fields, scrubs inherited env, spawns a child, handles Codex headless JSONL and Codex app-server output, posts callbacks, spools callbacks if HRC is unavailable, forwards signals, synthesizes degraded turn completion, and posts exit callbacks. It is highly useful as a seed, but not a clean broker yet because callback endpoints, HRC IDs, AgentChat registration, OTEL auth, and HRC environment variables are embedded.

  • exec.ts:18-29 — callback-or-spool sink.
  • exec.ts:182-201 — executable resolution.
  • exec.ts:227-337 — headless Codex JSONL pump.
  • exec.ts:339-415 — degraded/synthesized turn completion.
  • exec.ts:417-490 — Codex app-server one-shot driver integration.
  • exec.ts:617-640 — child spawn, env scrub, HRC env injection, stdio selection.
  • exec.ts:642-649 — signal forwarding.
  • exec.ts:651-727 — child-started, output drain, exit callback.
