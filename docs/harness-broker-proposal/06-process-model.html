Process Model

One-shot, resident, and daemon options with state machines and mode-specific behavior.

Recommendation

Use a one-shot broker per invocation as the default. Add a resident broker per runtime/invocation only where control after start is required. Do not start with a shared global daemon.

This matches the existing exec.ts launch model and HRC durability model while giving a clear path for SDK in-flight input and RPC/app-server harnesses.

Interactive mode

Preserve HRC/tmux ownership of the user-facing PTY initially. For interactive CLI launches, the broker should run inside the tmux pane and spawn the harness with ioMode: 'inherit' or equivalent. This keeps attach behavior and operator UX unchanged. Broker-owned PTY should be a later feature for non-tmux consumers.

SDK mode

The current Agent Spaces in-flight implementation stores a live UnifiedSession and send chain in memory. That proves true in-flight steering cannot be retrofitted to a completed one-shot process. For SDK-backed harnesses, use one-shot for simple non-interactive turns and resident broker for turns that advertise input.inFlight or need long-lived continuation state.

RPC/app-server mode

Codex app-server can be driven in one-shot mode for a single prompt, as exec.ts does today through runCodexAppServerOneShot. If later requirements include multi-turn RPC sessions, promotion to resident broker is straightforward: keep app-server child alive and accept invocation.input, invocation.interrupt, and invocation.stop commands.
