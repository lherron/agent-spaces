Event Model

Normalized events, correlation ownership, mapping from existing events, and replay rules.

Event envelope

Broker events should be correlation-light and sequence-heavy.

The broker should own invocationId, brokerSeq, event IDs, timestamps, driver kind, and provider/frontend facts. HRC should own host session IDs, run IDs, runtime IDs, generation, DB sequence, and stale-fence semantics. The broker may echo opaque correlation fields, but it must not interpret them.

Broker-owned vs HRC-owned fields

Field groupBroker-ownedHRC-owned
IdentityinvocationId, eventId, brokerSeq.launchId, runtimeId, runId, hostSessionId, generation.
TimingBroker event timestamp at emission.Persistence timestamp, HRC event stream ordering, replay timestamp if relevant.
Harness factsProvider, frontend, driver kind, child pid, io mode, capabilities, continuation observed from harness.Runtime transport, busy/ready/zombie state, launch record state, continuation persistence policy.
CorrelationOpaque echo only; useful for callback consumers.Authoritative. HRC validates generation/run/runtime fences and stores typed fields.
Raw provider payloadOptional, redacted, driver-specific raw data.Decides whether to store raw payload, derived semantic event, or both.

export interface BrokerEventEnvelope<TKind extends string, TPayload> {
  schemaVersion: 'broker.event.v1'
  eventId: string
  invocationId: string
  brokerSeq: number
  ts: string
  kind: TKind
  payload: TPayload
  correlation?: Record<string, string>   // echoed, not interpreted
  raw?: { source: string; value: unknown } // optional, redacted where needed
}

Ordering and replay rules

  1. Per invocation, brokerSeq is strictly monotonic. HRC can use eventId and brokerSeq for idempotent replay.
  2. Sink retries must not reorder a single invocation stream. Spool files should include seq in file name or record body.
  3. HRC can reorder only at persistence layer by HRC stream sequence. Broker sequence is a causal clue, not the global event order.
  4. Terminal events are explicit. turn.completed does not imply process exit; invocation.exited does not imply successful turn completion.
  5. Continuation is provisional until HRC policy accepts it. Broker reports observed continuation; HRC decides when to persist or clear stale continuation after failure.
