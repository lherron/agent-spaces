Input / Steering / Interrupt Split

Semantic definitions for user input, append context, steer/correction, interrupt, and stop.

Semantic split

HRC should request user-level intent; the broker should decide the provider-specific mechanism.

HRC should not know whether a steer maps to SDK interrupt(), Codex RPC cancel, Ctrl-C, or rejection. It should submit semantic input/control requests and record the broker’s accepted/rejected/effective-mode events.

Semantic actionMeaningHRC requestBroker decision
User inputA new user-authored turn/message.invocation.input(kind:'user') with policy such as idle-only, queue, or any.Send SDK prompt, start RPC turn, inject CLI input only if driver explicitly supports semantic CLI input; otherwise reject or rely on tmux user typing outside semantic API.
Append contextAdditional context that should not necessarily interrupt current reasoning.invocation.input(kind:'append_context').Native append if SDK supports it; coerce to queued user input only if declared safe; otherwise reject.
Correction / steerA higher-priority message intended to alter an active trajectory.invocation.input(kind:'steer', policy.conflict:'interrupt_then_send').Use native steer if available; else interrupt current turn and send a user message; else reject with unsupported capability.
InterruptAsk current turn to stop or yield, without necessarily ending invocation.invocation.interrupt(mode:'soft'|'cancel-turn'|'ctrl-c').Map to SDK interrupt, RPC cancel, SIGINT, pty Ctrl-C, or reject.
Stop / cancelEnd the invocation or runtime, possibly after grace period.invocation.stop(mode:'cancel-turn'|'terminate'|'kill').Attempt graceful SDK stop/RPC shutdown/SIGTERM then SIGKILL according to grace policy.

Capability-driven behavior

The compiler should include expected capabilities in the invocation spec. The broker should validate and may refine them at runtime. HRC should use capabilities to decide whether to offer in-flight controls to operators, but it should still handle rejection events because capabilities can be optimistic or version-dependent.

type InputCapability = 'unsupported' | 'idle-only' | 'in-flight' | 'native' | 'coerced-to-user' | 'interrupt-then-user'

interface InvocationCapabilities {
  input: {
    user: 'unsupported' | 'idle-only' | 'in-flight'
    appendContext: 'unsupported' | 'native' | 'coerced-to-user'
    steer: 'unsupported' | 'native' | 'interrupt-then-user'
  }
  interrupt: { soft: boolean; cancelTurn: boolean; signal: boolean }
  stop: { graceful: boolean; kill: boolean }
}

Idempotency

inputId and optional idempotencyKey should be part of every input request. The broker should emit input.accepted with disposition:'duplicate' for repeated accepted inputs, mirroring the current in-flight accepted-input set in run-tracker.ts.
