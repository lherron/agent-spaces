Recommendation

A concrete architecture proposal for introducing a harness-broker boundary between Agent Spaces harness normalization and HRC runtime supervision.
Strong recommendation

Build harness-broker as an Agent-Spaces-owned package first, with a protocol-only package that HRC consumes.

The cleanest near-term boundary is agent-spaces/packages/harness-broker plus agent-spaces/packages/harness-protocol. The broker should depend on existing Agent Spaces harness driver packages, but it must not depend on HRC. HRC should depend only on protocol types and the broker executable/package, not on Agent Spaces runtime methods.

A fully standalone monorepo is attractive only after the broker protocol stabilizes and the harness drivers are independently versioned. Doing it now would force premature extraction of Codex/Pi/Claude driver code and likely preserve today’s duplication under a new repo name.

Architecture in one page

ASP compiler/materializer
Resolve placement, choose harness/frontend, materialize bundle, prepare brain/tool env, produce compiled invocation.
→
HRC launch envelope
Adds runtime IDs, generation fence, callback/spool sink, supervision policy. Does not alter harness semantics.
→
Harness broker
Spawn/drive/control Codex, Pi, Claude Code, SDK sessions. Normalize events and continuation.
→
HRC runtime controller
Persist events, update run/runtime state, reject stale callbacks, expose operator/chat APIs.
