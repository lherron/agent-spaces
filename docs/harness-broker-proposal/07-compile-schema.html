The proposed HarnessInvocationSpec and HRC launch envelope split with examples.

Verdict on the proposed split

The split is correct, but make the invocation schema driver-oriented and keep HRC callback/spool/supervision out of it.

HarnessInvocationSpec should represent what to run and how the harness can be driven. HrcBrokerLaunchEnvelope should represent why HRC is launching it, how HRC wants to supervise it, and where events should go.

What belongs in HarnessInvocationSpec

BelongsReason
Provider, frontend, cwd, env, io mode, interaction modeThese are intrinsic to the compiled harness invocation.
Driver kind and driver-specific launch dataBroker needs to know whether it is spawning a plain process, driving Codex app-server, or creating an SDK session.
Continuation inputProvider-native resume state is part of harness invocation.
Initial prompt/attachments, if the driver needs them outside argvCodex app-server and SDKs need structured input. CLI may already have prompt encoded in argv.
CapabilitiesLets HRC know what control semantics are possible without provider-specific branching.
Display/audit metadataOptional, non-authoritative. Useful for logs/UI and debugging.

What does not belong in HarnessInvocationSpec

Does not belongWhere it belongs
launchId, runtimeId, runId, hostSessionId, generationHRC envelope and HRC persisted records.
Callback socket path, spool dir, HRC callback endpointHRC envelope / broker launch options.
Runtime DB state, fences, zombie/orphan metadataHRC runtime controller.
HRC-specific env like HRC_LAUNCH_ID, HRC_GENERATIONHRC envelope childEnv or callback sink config.
AgentChat registration targetsHRC launch/UI integration or a separate optional pre/post launch hook, not core broker spec.
Compile inputs such as placement, target composition, harness preferenceASP compiler input only.

Recommended TypeScript shape

export type HarnessProvider = 'anthropic' | 'openai' | (string & {})
export type HarnessFrontend =
  | 'claude-code'
  | 'codex-cli'
  | 'pi-cli'
  | 'agent-sdk'
  | 'pi-sdk'
  | (string & {})

export type HarnessContinuationRef = {
  provider: HarnessProvider
  key?: string
}

export type HarnessDriverSpec =
  | { kind: 'process'; argv: string[] }
  | {
      kind: 'codex-app-server'
      argv: string[]
      prompt?: string
      resumeThreadId?: string
      model?: string
      modelReasoningEffort?: string
      approvalPolicy?: 'untrusted' | 'on-failure' | 'on-request' | 'never'
      sandboxMode?: 'read-only' | 'workspace-write' | 'danger-full-access'
      profile?: string
      imageAttachments?: string[]
      featureFlags?: string[]
      extraArgs?: string[]
    }
  | {
      kind: 'agent-sdk'
      sessionId?: string
      model?: string
      plugins: Array<{ type: 'local'; path: string }>
      systemPrompt?: { content: string; mode: 'append' | 'replace'; sourcePath?: string }
      permissionMode?: 'auto' | 'prompt' | 'deny'
    }
  | {
      kind: 'pi-sdk'
      bundlePath: string
      sessionPath?: string
      model?: string
      yolo?: boolean
    }

export interface HarnessInvocationSpec {
  schemaVersion: 'harness.invocation.v1'
  provider: HarnessProvider
  frontend: HarnessFrontend
  cwd: string
  env: Record<string, string>
  ioMode: 'pty' | 'pipes' | 'inherit'
  interactionMode: 'interactive' | 'headless'
  driver: HarnessDriverSpec
  continuation?: HarnessContinuationRef
  initialInput?: {
    prompt?: string
    attachments?: Array<{ kind: 'file' | 'image'; path: string; mimeType?: string; filename?: string }>
  }
  capabilities?: InvocationCapabilities
  metadata?: {
    displayCommand?: string
    systemPromptFile?: string
    materializedBundle?: { path: string; targetName?: string; hash?: string }
    warnings?: string[]
    compiler?: { name: string; version?: string }
  }
}

export interface InvocationCapabilities {
  input: {
    user: 'idle-only' | 'in-flight' | 'unsupported'
    appendContext: 'native' | 'coerced-to-user' | 'unsupported'
    steer: 'native' | 'interrupt-then-user' | 'unsupported'
  }
  interrupt: {
    soft: boolean
    cancelTurn: boolean
    signal: boolean
  }
  stop: {
    graceful: boolean
    kill: boolean
  }
  events: {
    assistantDelta: boolean
    assistantMessage: boolean
    toolCall: boolean
    toolResult: boolean
    usage: boolean
    continuation: boolean
  }
}

HRC envelope shape

export interface HrcBrokerLaunchEnvelope {
  schemaVersion: 'hrc.broker-launch.v1'
  launchId: string
  runtimeId: string
  runId?: string
  hostSessionId: string
  generation: number

  invocation: HarnessInvocationSpec

  callback?: {
    kind: 'unix-http'
    socketPath: string
    endpointBase: `/v1/internal/broker/${string}` | string
    authHeader?: { name: string; value: string }
  }
  spool?: { dir: string }

  // HRC IDs echoed by broker only; authoritative interpretation stays in HRC.
  correlation?: Record<string, string>

  // Explicit host/runtime env additions. These are not part of the compiled harness spec.
  childEnv?: Record<string, string>

  lifecycleAction?: 'attach' | 'start' | 'turn'
  supervision?: {
    processModel: 'oneshot' | 'resident'
    waitForCompletion?: boolean
    signalForwarding?: Array<'SIGHUP' | 'SIGTERM' | 'SIGINT'>
  }
  display?: {
    printHeader?: boolean
    printCommand?: boolean
    prompts?: {
      system?: { content: string; mode?: 'append' | 'replace'; deliveredVia?: string; sourcePath?: string }
      priming?: { content: string; deliveredVia?: string }
    }
  }
  hookBridge?: unknown
  otel?: unknown
}

Compatibility with today’s shape

The current proposed minimal split in the prompt is a good first migration step:

type HarnessInvocationSpec = {
  provider: string
  frontend: string
  argv: string[]
  env: Record<string, string>
  cwd: string
  ioMode: 'pty' | 'pipes' | 'inherit'
  interactionMode?: 'interactive' | 'headless'
  continuation?: { provider: string; key?: string }
  driver?: { kind: 'codex-app-server'; /* ... */ }
}

type HrcBrokerLaunchEnvelope = {
  launchId: string
  runtimeId: string
  runId?: string
  hostSessionId: string
  generation: number
  callbackSocketPath: string
  spoolDir: string
  invocation: HarnessInvocationSpec
}

For implementation, prefer the richer driver-oriented shape above because it avoids overloading top-level argv. If backward compatibility matters, accept both forms during migration and normalize internally to driver.kind.
