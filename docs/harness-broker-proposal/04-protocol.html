Broker Protocol

A JSON-RPC/NDJSON command protocol with request/response interface sketches.

Transport recommendation

Use JSON-RPC 2.0 messages over NDJSON for control, plus a callback/sink abstraction for events.

JSON-RPC over NDJSON is easy to drive from HRC over stdio or a Unix socket, works for resident brokers, and preserves request/response semantics for input/interrupt/stop. One-shot launches can still use an envelope file and event callback sink without requiring a control connection.

Two entrypoints

EntrypointUseShape
harness-broker run --envelope <file>One-shot launch replacing exec.ts. Best for tmux interactive, detached headless CLI, and one-shot app-server.Reads envelope, starts invocation, emits callbacks/spool/stdout events, exits when child/session exits.
harness-broker serve --stdio or --socket <path>Resident broker per runtime/invocation when HRC needs control after start: SDK in-flight input, RPC app-server multi-turn, long-lived headless.HRC sends JSON-RPC commands; broker emits broker.event notifications or posts to configured sink.

Command list

CommandPurposeNotes
broker.helloProtocol/version/capability negotiation.First command in resident mode. Not required for one-shot envelope mode.
broker.healthHealth and optional invocation summary.Used by HRC supervision and diagnostics.
invocation.startStart a compiled invocation.Accepts HarnessInvocationSpec; no placement or compile input.
invocation.inputSubmit user input, append context, or steering correction.Returns accepted/rejected/queued/coerced result with effective semantics.
invocation.interruptInterrupt current turn without necessarily ending invocation.Broker maps to SDK interrupt, RPC cancel, SIGINT, pty Ctrl-C, or rejects.
invocation.stopEnd the turn/invocation/runtime.Graceful cancel/terminate/kill policy.
invocation.statusSnapshot state, continuation, pid, current turn, pending inputs.Useful after HRC reconnects to resident broker.
invocation.disposeRelease broker state after terminal invocation.Optionally kills still-running process if HRC is abandoning it.
Excluded commands: no compile, materialize, resolvePlacement, buildProcessInvocationSpec, chooseHarness, or prepareEnv commands. Those are compiler/materializer responsibilities.

TypeScript command sketch

export type JsonRpcId = string | number

export type BrokerCommand =
  | BrokerHelloRequest
  | BrokerHealthRequest
  | InvocationStartRequest
  | InvocationInputRequest
  | InvocationInterruptRequest
  | InvocationStopRequest
  | InvocationStatusRequest
  | InvocationDisposeRequest

export interface JsonRpcRequest<TMethod extends string, TParams> {
  jsonrpc: '2.0'
  id: JsonRpcId
  method: TMethod
  params: TParams
}

export interface JsonRpcResponse<TResult = unknown> {
  jsonrpc: '2.0'
  id: JsonRpcId
  result?: TResult
  error?: { code: string; message: string; data?: unknown }
}

export type BrokerHelloRequest = JsonRpcRequest<'broker.hello', {
  client: { name: string; version?: string }
  protocolVersion: '2026-05-20'
  capabilitiesWanted?: string[]
}>

export type BrokerHelloResponse = {
  broker: { name: 'harness-broker'; version: string }
  protocolVersion: '2026-05-20'
  capabilities: BrokerCapabilitySet
}

export type BrokerHealthRequest = JsonRpcRequest<'broker.health', {
  includeInvocations?: boolean
}>

export type BrokerHealthResponse = {
  status: 'ok' | 'degraded' | 'draining'
  uptimeMs: number
  invocations?: Array<InvocationStatus>
  warnings?: Array<{ code: string; message: string }>
}

export type InvocationStartRequest = JsonRpcRequest<'invocation.start', {
  invocationId?: string
  spec: HarnessInvocationSpec
  launch?: BrokerLaunchOptions
  correlation?: Record<string, string>
}>

export type InvocationStartResponse = {
  invocationId: string
  state: InvocationState
  capabilities: InvocationCapabilities
  pid?: number
  pty?: { cols: number; rows: number }
}

export type InvocationInputRequest = JsonRpcRequest<'invocation.input', {
  invocationId: string
  inputId: string
  kind: 'user' | 'append_context' | 'steer'
  content: BrokerInputContent
  policy?: {
    when?: 'idle' | 'in_flight' | 'any'
    conflict?: 'reject' | 'queue' | 'interrupt_then_send'
    idempotencyKey?: string
  }
}>

export type InvocationInputResponse = {
  accepted: boolean
  disposition: 'sent' | 'queued' | 'coerced' | 'rejected' | 'duplicate'
  effectiveKind?: 'user' | 'append_context' | 'steer'
  pendingInputs?: number
  reason?: string
}

export type InvocationInterruptRequest = JsonRpcRequest<'invocation.interrupt', {
  invocationId: string
  interruptId?: string
  mode?: 'soft' | 'ctrl-c' | 'cancel-turn'
  reason?: string
}>

export type InvocationInterruptResponse = {
  accepted: boolean
  effectiveMode?: 'sdk-interrupt' | 'rpc-cancel' | 'sigint' | 'pty-ctrl-c' | 'unsupported'
  state: InvocationState
  reason?: string
}

export type InvocationStopRequest = JsonRpcRequest<'invocation.stop', {
  invocationId: string
  stopId?: string
  mode?: 'cancel-turn' | 'terminate' | 'kill'
  graceMs?: number
  reason?: string
}>

export type InvocationStopResponse = {
  accepted: boolean
  effectiveMode: 'cancel-turn' | 'sigterm' | 'sigkill' | 'sdk-stop' | 'unsupported'
  state: InvocationState
}

export type InvocationStatusRequest = JsonRpcRequest<'invocation.status', {
  invocationId: string
}>

export type InvocationStatusResponse = InvocationStatus

export type InvocationDisposeRequest = JsonRpcRequest<'invocation.dispose', {
  invocationId: string
  kill?: boolean
}>

export type InvocationDisposeResponse = { disposed: true }

Supporting types

export type InvocationState =
  | 'created'
  | 'starting'
  | 'running'
  | 'idle'
  | 'turn-running'
  | 'interrupting'
  | 'stopping'
  | 'exited'
  | 'failed'
  | 'disposed'

export interface BrokerLaunchOptions {
  mode: 'oneshot' | 'resident'
  eventSink?:
    | { kind: 'stdio' }
    | { kind: 'unix-http'; socketPath: string; endpoint: string; authHeader?: { name: string; value: string } }
    | { kind: 'file-spool'; dir: string; forward?: { kind: 'unix-http'; socketPath: string; endpoint: string } }
  childEnv?: Record<string, string>       // host/runtime-specific env additions; explicit only
  inheritedEnvPolicy?: 'scrub-default' | 'empty' | { allow: string[]; denyPrefix?: string[] }
  signalPolicy?: { forward: Array<'SIGHUP' | 'SIGTERM' | 'SIGINT'>; exitAfterForward?: boolean }
  display?: { printHeader?: boolean; printCommand?: boolean }
}

export interface InvocationStatus {
  invocationId: string
  state: InvocationState
  provider: string
  frontend: string
  driverKind: string
  pid?: number
  currentTurnId?: string
  pendingInputs: number
  brokerSeq: number
  startedAt?: string
  lastActivityAt?: string
  continuation?: HarnessContinuationRef
  capabilities: InvocationCapabilities
  exit?: { exitCode?: number; signal?: string; reason?: string }
}

export interface BrokerInputContent {
  text?: string
  attachments?: Array<{ kind: 'file' | 'image'; path: string; mimeType?: string; filename?: string }>
  metadata?: Record<string, unknown>
}

Event delivery modes

ModeWhen to useReliability model
stdioDirect headless subprocess where HRC waits and reads broker stdout.HRC persists immediately; no extra callback socket. Best for synchronous runs.
unix-httpOne-shot launched inside tmux or detached headless where HRC cannot own stdout.Broker posts events to HRC; HRC handles fence and persistence.
file-spool + forwardSame as current exec.ts callback/spool reliability.Broker writes undelivered events with event IDs/seq. HRC owns replay and stale rejection.

Error response conventions

CodeMeaning
invalid_specCompiled invocation missing required fields or incompatible driver/mode.
unsupported_capabilityRequested input/interrupt/stop semantic cannot be represented by the driver.
invocation_not_foundControl command references unknown or disposed invocation.
invalid_stateCommand is valid but not in the current invocation/turn state.
spawn_failedExecutable resolution or child process start failed.
driver_failedCodex app-server/SDK driver failed after launch.
sink_unavailableEvent sink unavailable and no spool configured.
