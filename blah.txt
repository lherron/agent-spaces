HRC runtime architecture
Separate runtime intent from runtime control.
The codebase already treats provisioning and runtime control as different decisions. The improvement path is to make that separation a first-class model instead of a set of predicates and stringly runtime rows spread across the server.

The separation exists, but it is implicit.
Requests describe placement, model provider, and desired harness. Runtime records, run records, lifecycle events, and responses report HRC's selected runtime controller. The split is visible in contracts, adapters, persistence, and transport managers.

Type Evolution
As-is and to-be, split by type.
The current vocabulary mixes product intent, invocation mechanics, runtime lanes, and state payloads. The proposed model renames provider intent to modelProvider, splits requested harness from materialized harness runtime, tightens runtime control into an HRC-owned controller, and keeps runtimeState as the missing type that makes invalid combinations unrepresentable.

Decision: sdk and headless are not controllers
Today, sdk means HRC is running an Agent Spaces SDK turn in process, while headless means the runtime is not attachable and may be backed by either SDK or CLI execution. In the proposed vocabulary, SDK is expressed by harnessRuntime, startupMethod, or turnDelivery; headless is expressed by interactionMode.

Recommendations
Make the existing boundary executable.
The goal is not a large rewrite. The path is to name the hidden decision points, type the impossible states out of existence, and let server modules reflect the architecture already present in the contracts.

1
Extract a RuntimeRouteDecision module.
Centralize the current predicates into a pure routing function that returns an explicit decision: model provider, agent harness, harness runtime, runtime controller, interaction mode, startup method, turn delivery, reuse strategy, and whether provisioning must normalize the harness to interactive.

Most important because drift here affects runtime identity, reuse, attach behavior, and whether a turn goes through literal terminal input, SDK execution, a broker invocation, or a managed CLI launch.
2
Rename transport to runtimeController.
Keep the HRC-owned field for runtime lifecycle/control semantics only. Remove sdk and headless from controller values; model them as harnessRuntime, interactionMode, and turnDelivery.

This answers the current ambiguity directly: SDK is how a harness runs; headless is how it interacts with the host.
3
Use discriminated runtime state.
Replace opaque optional tmuxJson and surfaceJson checks with runtime-controller-specific runtime variants. Make it impossible to represent runtimeController: 'tmux' without tmux pane metadata.

This reduces defensive runtime checks and makes attach/capture/literal-send capabilities a property of the type, not repeated conditionals.
4
Rename the Agent Spaces contract nouns.
Move provider toward modelProvider and frontend toward harnessRuntime in Agent Spaces first, then mirror the aliases in HRC during migration.

Agent Spaces owns placement, harness catalog, model resolution, and invocation specs; HRC should consume those names rather than re-invent them.
5
Split the server file around actual responsibilities.
Extract small modules for routing, interactive runtime provisioning, SDK dispatch, managed child-process execution, launch artifact building, startup, turn delivery, and controller operations.

The system is conceptually modular already; file boundaries should make that clear.
Re-imagined
Runtime controller names HRC control.
A small vocabulary shift would remove the biggest ambiguity. Agent Spaces owns the requested harness and materialized harness runtime. HRC owns runtime controller, runtime state, reuse, and turn delivery.

type AgentRuntimeIntent = {
  placement: RuntimePlacement
  modelProvider: 'anthropic' | 'openai'
  agentHarness?: 'claude-code' | 'codex' | 'pi'
  preferredHarnessRuntime?: HarnessRuntime
  interactionMode?: 'interactive' | 'headless' | 'nonInteractive'
}
Cross-repo target: Agent Spaces owns these request nouns; HRC mirrors them in intent.
